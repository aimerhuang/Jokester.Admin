# 后台管理系统设计书

## 1. 项目概述

### 1.1 项目背景

当前需要建设一套基于 **.NET 10 Web API** 的后台管理系统，用于统一管理多个业务站点，并实现用户、页面、功能权限的集中控制。系统采用前后端分离架构，后端负责认证、授权、业务处理与数据管理，前端负责界面展示与交互。

现有业务站点包括：

1. 个人博客网站：支持文章增删改查、发布管理
2. GPT 生图网站：管理生图功能，无权限用户不可执行生图操作
3. 未来新增站点：后台需支持快速扩展与接入

### 1.2 项目目标

建设一套可扩展的后台管理平台，实现：

- 多站点统一管理与入口
- 用户统一登录认证（JWT）
- 页面级、按钮级、接口级三层权限控制
- 支持后续新增站点、页面、功能权限
- 支持灵活的用户、角色、权限分配
- 完整的操作审计能力

### 1.3 项目定位

本系统定位为统一权限与后台运营管理平台，既是各网站的后台入口，也是权限中心和资源管理中心。

## 4. 权限模型设计

### 4.1 权限模型

采用 **RBAC + 站点维度** 模型：

- 用户绑定角色，角色绑定菜单/按钮/接口权限
- 用户绑定可访问站点
- 超级管理员（`is_super_admin=1`）绕过所有权限校验

### 4.2 权限控制三层

| 层级 | 控制粒度 | 示例 |
|------|----------|------|
| 站点级 | 用户是否可进入某网站后台 | 可访问博客后台，不可访问 GPT 生图后台 |
| 页面级 | 用户是否可访问某页面 | 可进文章管理页，不可进用户管理页 |
| 操作级 | 用户是否可执行具体功能 | 可新增文章，不可删除文章 |

### 4.3 权限编码规范

格式：`系统.模块.动作`

```text
Blog.Article.View
Blog.Article.Create
Blog.Article.Update
Blog.Article.Delete
Blog.Article.Publish

Blog.Media.View

Blog.Dashboard.View

AiImage.Page
AiImage.Generate

```

## 5. 业务流程设计

### 5.1 登录流程

1. 用户提交账号密码
2. 后端校验账号状态与密码
3. 生成 AccessToken + RefreshToken
4. 返回用户信息、可访问站点、权限列表
5. 前端存储 Token 并渲染菜单

### 5.1.1 注册邮箱验证码流程

1. 前端调用 `POST /api/auth/register/email-code`，请求体传 `email`
2. 后端执行邮箱本地基础验证：格式、长度、黑名单域名
3. 当 `EmailValidation.EnableApiValidation=true` 时，后端额外调用第三方邮箱验证 API；为 `false` 时跳过第三方验证
4. 后端生成 6 位验证码，按规范化后的邮箱写入 Redis，有效期 10 分钟
5. 后端通过 `Mail` 配置的 SMTP 服务发送验证码邮件
6. 前端调用 `POST /api/auth/register`，请求体传 `userName`、`nickName`、`password`、`email`、`emailCode`
7. 后端用同一个 `email` 查询 Redis 验证码，校验通过后创建用户并删除验证码键

### 5.2 Token 刷新策略

- AccessToken：2 小时
- RefreshToken：7 天
- RefreshToken 默认存储于 Redis，可主动失效

### 5.3 权限校验流程

1. 请求进入
2. JWT 中间件校验 Token
3. 权限中间件读取接口声明的 `permission_code`
4. 优先从 Redis 读取用户权限
5. 不命中时回退数据库
6. 不具备权限则返回 403

### 5.4 多站点切换流程

1. 登录后获取可访问站点列表
2. 前端按站点显示导航
3. 切换站点后加载该站点菜单树
4. 业务表使用 `site_id` 标记归属

### 5.5 博客站点固定路由约束

当前博客后台虽然沿用站点模型，但接口层按“单博客站点”收敛：

- 博客文章和媒体接口固定归属 `siteCode=blog`
- 文章新增/编辑、文章列表、媒体上传、媒体列表均不再要求调用方传 `siteId`
- 后端在服务层解析 `blog` 站点 ID 后，再执行数据写入和查询过滤
- 数据表中的 `site_id` 仍保留，作为数据归属字段和后续多站点扩展基础

### 5.6 博客媒体与缩略图流程

1. 上传媒体写入 `blog_media`，返回可插入正文的 `url`
2. 创建或更新文章时，后端解析正文 HTML 中的 `<img src="...">`
3. 解析出的 URL 匹配 `blog_media.url` 后，写入 `blog_article_media`
4. 文章列表和详情返回 `thumbnailUrl`，优先级为 `cover_url`、文章媒体关联第一张图、正文第一张图
5. 文章正文仍返回 HTML 内容，由前端按 HTML 渲染图片和段落

## 7. 数据库表结构设计

### 7.1 `sys_user`

用户基础信息、登录状态、超级管理员标记。

### 7.2 `sys_role`

角色信息与角色编码，如 `blog_admin`。

### 7.3 `sys_site`

站点信息与站点编码，如 `blog`、`ai_image`。

### 7.4 `sys_menu`

菜单、按钮、接口权限统一建模，按 `site_id` 归属站点。

### 7.5 关联表

- `sys_user_role`
- `sys_role_menu`
- `sys_user_site`
- `blog_article_media`

### 7.6 `blog_article`

博客文章表，关键字段：

- `site_id`
- `title`
- `summary`
- `content`
- `cover_url`
- `tags`
- `status`：0 草稿，1 发布，2 隐藏
- `view_count`

说明：

- 当前实现仍保留 `site_id` 外键
- 但博客后台接口不再让前端传 `siteId`
- 文章新增/编辑接口允许通过 `status` 设置状态
- `site_id` 由后端根据 `siteCode=blog` 自动写入
- 列表和详情接口返回 `coverUrl` 与 `thumbnailUrl`
- `thumbnailUrl` 优先使用 `cover_url`，其次使用 `blog_article_media` 关联的第一张媒体图，最后从正文第一张图片兜底提取

### 7.7 `blog_media`

博客媒体资源表，关键字段：

- `site_id`
- `file_name`
- `storage_key`
- `url`
- `mime_type`
- `file_size`
- `storage_provider`

说明：

- 当前实现仍保留 `site_id` 外键
- 上传和列表查询均固定绑定 `blog` 站点

### 7.8 `blog_article_media`

文章媒体关联表，用于追踪文章正文引用了哪些已上传媒体资源：

- `article_id`
- `media_id`
- `created_at`

说明：

- 创建或更新文章时由后端根据正文 `<img src="...">` 自动同步
- 仅当图片 URL 能匹配 `blog_media.url` 时写入关联
- 缩略图逻辑会优先读取该表关联的第一张媒体图

### 7.9 `ai_image_parameter`

生图参数表，按 `param_type` 区分：

- `resolution`：分辨率档位，`1k/2k/4k` 对应长边 `1024/2048/3840`；实际输出尺寸还会按供应商限制压到 `16px` 倍数且总像素不超过 `8,294,400`
- `quality`：质量档位，`low/med/high`，其中 `med` 的供应商参数为 `medium`
- `aspect_ratio`：画幅比例，支持 `1:1`、`16:9`、`9:16`、`4:3`、`3:4`、`3:2`、`2:3`、`21:9`

### 7.10 `ai_image_task`

生图任务记录表，按 `site_id` 标识归属。任务保存用户选择的 `resolution_code`、`quality_code`、`aspect_ratio_code`，创建任务时计算出的 `width`、`height`、`size`、供应商 `quality` 快照，以及可选参考图地址集合 `reference_image_urls`。

### 7.11 `sys_login_log`

登录日志。

### 7.12 `sys_operation_log`

操作日志。

## 8. 表关系设计

### 8.1 核心关系

- `sys_user` -> `sys_user_role` -> `sys_role`

- `sys_user` -> `sys_user_site` -> `sys_site`

- `blog_article` -> `blog_article_media` -> `blog_media`

### 9.8 积分与 AI 生图接口

| 方法 | 路径 | 权限码 | 说明 |
|------|------|--------|------|
| GET | `/api/points/balance` | 登录用户 | 查询当前用户积分余额和今日签到状态 |
| POST | `/api/points/sign-in` | 登录用户 | 每日签到领取 25 积分；同一自然日只能成功一次 |
| GET | `/api/ai/images` | `AiImage.Page` | AI 生图记录分页 |
| GET | `/api/ai/images/models` | `AiImage.Generate` | 获取启用的 AI 图片模型 |
| GET | `/api/ai/images/parameters` | `AiImage.Generate` | 获取分辨率、质量、画幅比例参数选项和启用的积分价格 |
| POST | `/api/ai/images/parameters/resolve` | `AiImage.Generate` | 将 `resolutionCode`、`qualityCode`、`aspectRatioCode` 解析为实际 `width`、`height`、`size` 和供应商质量参数 |
| POST | `/api/ai/images` | `AiImage.Generate` | 创建 GPT Image2 后台生图任务，记录 `prompt`、参数编码、计算后的宽高、`size`、`quality`、`imageCount` 和可选参考图 URL |
| POST | `/api/ai/images/generate` | `AiImage.Generate` | 直接生成 GPT Image2 图片，返回静态图片 `url`、参数编码和计算后的宽高 |
| POST | `/api/ai/images/nanoBananaImage/generate` | `AiImage.Generate` | 直接生成 Nano Banana2 图片；不传 `imageUrls` 时文生图，传 `imageUrls` 时图生图 |
| POST | `/api/ai/images/nanoBananaImage` | `AiImage.Generate` | 创建 Nano Banana2 后台生图任务 |
| POST | `/api/ai/images/upload` | 登录用户 | 上传 AI 图片引用文件 |
| GET | `/api/ai/images/{id}` | `AiImage.Record.View` | 详情 |
| POST | `/api/ai/images/{id}/favorite` | 登录用户 | 收藏或取消收藏结果图片 |
| DELETE | `/api/ai/images/{id}` | `AiImage.Record.Delete` | 删除记录 |

注册成功自动赠送 50 积分。生图前按 `ai_image_point_price` 查询价格并扣除 `points * imageCount`；GPT Image2 按 `modelCode + resolutionCode + qualityCode` 匹配，Nano Banana2 官方无 `quality` 参数，只按 `modelCode + resolutionCode` 匹配，`quality_code` 列不参与查询（存 `''` 或 `NULL` 都不影响）。任务失败或超时时写入积分返还流水。

分辨率档位按长边计算：`1k=1024`、`2k=2048`、`4k=3840`。实际输出尺寸会按供应商限制压到 `16px` 倍数且总像素不超过 `8,294,400`；例如 `4k + med + 1:1` 会解析为 `2880x2880`，`4k + med + 16:9` 会解析为 `3840x2160`，供应商质量参数为 `medium`。

GPT Image2 生成图片保存到服务器静态目录 `wwwroot/ai-images/{yyyyMM}/`。请求体可选 `referenceImageUrls`，最多 6 张；前端应先上传参考图到后端并传内部 URL JSON 数组。有参考图时后端调用 OpenAI `/images/edits`，把内部 URL 解析为 `wwwroot` 下的本地文件，并以 multipart `image[]` 文件字段提交；无参考图时调用 `/images/generations`。Nano Banana2 生成图片保存到服务器静态目录 `wwwroot/nano-banana2-images/{yyyyMM}/`。后台任务完成后，`ai_image_task.result_urls` 保存生成图片 URL 的 JSON 数组，列表和详情接口返回 `resultUrls`；`ai_image_task.reference_image_urls` 保存参考图 URL 的 JSON 数组。

Nano Banana2 配置项通过 `.env` / 环境变量注入：`NanoBanana2__BaseUrl`、`NanoBanana2__ApiKey`、`NanoBanana2__ImageModel`、`NanoBanana2__TextToImagePath`、`NanoBanana2__ImageToImagePath`。

## 12. 扩展性设计

### 12.1 新增站点流程

1. 在 `sys_site` 新增站点
2. 在 `sys_menu` 配置站点菜单和权限节点
3. 为用户分配站点访问权限
4. 为角色分配菜单权限
5. 新增业务表和业务接口，并继续保留 `site_id`

### 12.2 从单博客接口恢复多站点

如果后续博客模块需要重新支持多站点，有两种可行路线：

- 在现有服务层恢复显式 `siteId` / `siteCode` 入参
- 新增更通用的站点化博客接口，如 `/api/sites/{siteCode}/blog/articles`

当前不建议直接删掉 `site_id` 列，因为这会抹掉未来的扩展基础。

## 15. 结论

本系统采用 `.NET 10 Web API + Vue 3 + MySQL + Redis + JWT` 技术栈，以 `RBAC + 站点维度` 作为统一权限模型。

当前博客模块的取舍是：

- 数据层保留 `site_id`
- 接口层默认固定走 `siteCode=blog`
- 先满足当前“单博客站点后台”的简化需求
- 同时保留未来恢复多站点能力的空间
